# 1 算子设计(mskpp)功能域概述

名称：算子设计(mskpp)

描述：以算子设计为中心，通过算子实际运行和建模仿真方法，实现对算子的性能评估，辅助算子设计。

在算子实际运行方面，配合高级语言的能力，通过对参数组合空间的搜索，配合算法，减少搜索空间，从而选择出最优的参数，达成设计要求。

在建模仿真方面，通过对指令建模，配合高效的调度能力，实现建模的底层能力。用户通过编程语言表达计算逻辑，最终实现对算子的性能的理论评估。

# 2 算子设计(mskpp)功能域总体方案

算子设计本身包含预测，评估等一系列行为以及为了这一系列行为的辅助工作。随着开发方式的丰富，除了基于Ascend C语言的从0开发以外，还存在模版库ACTLASS等开发方式。所以，对于算子设计的诉求也产生了变化：除了支撑评估分析数学公式算子化后的性能情况外，还需要支持对模版参数的选择和评估。

对于上述算子实现和对算子设计要求，我们总结为2种：1. 在高度自由度下的公式实现 -- 评估数学公式实例化后的性能；2. 在可选参数组合下的最优组合选择 -- 提供自动选择最优参数的能力

业内方案洞察

在AI发展后，相对于原普通的编译器，提出了AI编译器的概念，将更多的上下文信息传递到编译器中，从而实现更优的参数选择，而由于各种策略本身存在无法共存的情况，所以，这类语言提出了自动调优的概念。为了保证评估的准确性，一般使用真实硬件上运行进行评估。

比较Triton和TVM的方案，整体结论如下：

\1. 无论哪种方案，本质上都需要定义参数搜索空间。从编程简易性来讲，Triton的方案较为友好，但是TVM提供了搜索算法来简化自动的过程。

\2. AI编译器本身会内含kernel运行的能力，将编译-运行的过程隐藏到编译器内部，从而较为便捷地进行自动寻优。

基于建模

昇腾芯片上存在多级缓存，在算子开发过程中，需要考虑存储与计算的分配关系，并依据算法的bound选择合适的tiling策略。而现有昇腾profiling不支持指令级的度量，对于昇腾算子的性能调试只能黑盒进行。而，使用cycle级仿真器，则存在仿真时间极长，且对于规模大于一定程度的算子，会遇到内存不足等无法仿真的情况。为了使得**算子开发人员快速找到最优****tiling****策略**，开发了算子理论性能预测工具。

该功能可以在算子实现前辅助分析tiling策略，从而快速找到较好的tiling策略。也可以用于分析算子性能瓶颈：由于现有profiling工具无法感知比kernel粒度更小的指令的调度情况，通过该方法可以快速了解kernel内部的瓶颈。

对于特定的数学表达是无法直接用于预测性能的，因为数学表达的具体实现有着多种算法。以经典的<算法导论>中的排序算法为例，为了实现数据的排序，本身存在多种复杂度不同的算法。所以，对于算子理论性能预测工具而言，需要以数学表达的特定算法实现作为输入，我们需要设计一套简洁的DSL语言。

对于算子，其本身由一条条指令组成，不同指令由TS调度完成。为了仿真这一过程，我们引入离散事件系统，实现对于指令的调度，我们会以0等待实现调度，符合“理论”两字。同时，每条指令调度的具体性能参数由实测获得（PS: 通过构造单指令的算子，实现对单指令能力的度量）。

在DSL语言方面，考虑到本身这个模型仅针对性能，换句话说，不用考虑功能的正确性，故而，其接口设计和使用方法可以非常简洁，能在 10s 内获得算子的性能结果。

算子理论性能预测工具是一个开放式的工具，本身支持用户进行DSL前端的注册和指令性能数据的注册，并提供建模辅助工具帮助用户完成指令性能建模。整体架构如下：

### 4.1.3 实现思路

开发者完成kernel代码开发后，希望直接进行kernel测试，不关注kernel输入与调用如何构造，在昇腾算子开发流程中，开发者要kernel测试一般需要：

\1. 为算子写torch_npu的适配层接入PyTorch，利用PyTorch API完成算子输入输出的构造与调用，或

\2. 利用ACL系列的C接口完成算子输入输出的构造，通过ACLNN或<<<>>>调用符进行算子调用，或

\3. 其他测试框架辅助完成上述过程

上述过程较为繁琐且不是算子开发者的关注点，同时模板库的引入使tiling参数调试变为排列组合选择问题，要求开发者反复编译测试，业界在该场景下的调试体验例如Triton可直接利用PyTorch API创建输入输出tensor传递给算子，算子可通过特定DSL在Python完成调用，无需开发者额外适配，虽然Triton的算子实现本身在Python DSL中，昇腾算子实现是AscendC，但开发者进行算子测试的便利性流程设计是可以借鉴的，mskpp沿用这个设计提供相同便利性的Python DSL向开发者提供算子上板测试接口，大体分以下几个步骤，参考下图所示：

\1.    利用PyTorch API构造算子输入输出tensor；

\2.    指定算子实现代码路径与入口函数，完成算子信息配置；

\3.    生成算子下发代码，完成编译；

\4.    算子下发调用。

### 4.1.4 实现设计

#### 4.1.4.1 支持kernel运行

为使算子可通过mskpp提供的DSL运行，需要DSL实现以下的功能点：

\1.    配置算子的源码信息与kernel入口，

\2.    生成算子下发调用代码，

a.    生成临时host侧代码，提供对应的Python接口关联算子下发C++接口，支持外部框架如PyTorch构造的tensor作为输入；

b.    生成编译脚本；

\3.    完成算子代码与下发代码的编译，支持被Python导入；

\4.    配合参数空间搜索功能，

a.    支持修改并生成临时算子代码；

b.    算子调用时支持接入其他算子工具；（算子下发接口可被劫持）

c.    支持接入算子性能评估；（利用Profiling接口对算子耗时进行采集）

 

代码生成

算子下发代码生成在算子调用前完成，开发者的输入仅有代码路径与算子入口函数。

下发代码生成主要考虑以下几个设计点：

\1.    算子launch形式，建议使用较为简洁的<<<>>>调用符实现，相比直接调用runtime接口或新生成aclnn接口，可降低生成代码的复杂度；

\2.    算子入口函数入参个数。算子launch入参传递相关的代码生成时，因入参由Python侧传入，C++代码中进行解析，算子入参对象需要动态生成，其依赖获得入参个数信息，入参个数需要对代码进行分析获得，分析思路可参考以下几种方式：

a.    使用get_kernel函数上一层调用栈的入参个数（需约束开发者使用接口的调用方式，方案简单）

b.    算子代码中匹配入口函数找到函数定义位置，计算()括号作用域内的参数个数；

\3.    Python代码中使用C++扩展的实现方式。为获得更高的性能建议采用Direct C API (#include <Python.h>)的方法，此方法要求编译下发代码为一个module导入Python中。

\4.    CPU内存与GM之间的搬运。一般kernel函数入参均为GM内存指针，对于框架创建的tensor，Python侧需获取其内容指针传给C++接口，生成代码完成GM内存申请，tensor数据从CPU内存到GM的拷贝，算子运行结束后拷贝GM内容回CPU内存。

自动编译

临时编译脚本基于模板库算子编译脚本，新增实现以下功能：

\1.    -I增加Python头文件目录；

\2.    生成产物由executable变为动态链接库的形式，以作为extension module导入Python；

\3.    保持编译产物对runtime的动态链接以支持与其他算子工具协同。

kernel性能度量

关注算子本身在device侧的运行时间，可采用轻量化的MSPTI的Python接口 KernelMonitor。

下发代码与算子代码一起编译，获得算子对象，提供下发接口，下发接口形式采用 kernel[blockdim](a, b, c)，由KernelInterface实现。

```Python
class KernelInterface(Generic[T]): 
   """ 为算子提供形如 kernel[blockdim](x, y, z) 的调用形式 """ 
   launch: T 
   def __getitem__(self, blockdim=None, l2ctrl=None, stream=None) -> T: 
     """ 
     A kernel function is invoked with: fn[blockdim, l2ctrl, stream](*args, **kwargs) 
     """ 
     return lambda *args, **kwargs: self.launch(blockdim=blockdim, l2ctrl=l2ctrl, stream=stream, 
                         warmup=False, *args, **kwargs) 
 class CompiledKernel(KernelInterface[T]): 
   def __init__(self, config: Config): 
     pass 
   def launch(self, blockdim, l2ctrl=None, stream=None, warmup=None, *args, **kwargs): 
     ''' 算子下发 ''' 
     pass 
 class Compile(): 
   def __init__(self, config: Config): 
     self.config = config 
     pass 
   def do_compile(self): 
     return CompiledKernel(self.config) 
 def compile(config: Config): 
   ''' 完成编译，返回支持下发的算子对象 ''' 
   return Compile(config).do_compile()
```

   src = mskpp.Launcher(config).code_gen() # 生成下发代码

   kernel = get_kernel(src_path, kernel_name) 
   blockdim = 16 
   kernel[blockdim](a, b, c) # 算子调用 
   return c

### 4.2.3 实现思路

![img](../pic/d413dede75337df62260a66db3bdff09_130x392.gif@900-0-90-f.gif)

模板库的引入使GEMM运算的tiling调试参数分解为C++抽象模板实现的排列组合，模板原语支持通过自定义数据类型、shape大小等形式灵活地调整tiling参数，如此可以使开发者轻松地定制自定义GEMM kernel。

\1. tiling参数以排列组合形式统一通过模板传入kernel，可寻优参数的识别与替换就变得较为容易。

\2. 在参数集中确定最优参数，还需要定义搜索空间，以业界Triton举例，其支持开发者通过两种方式定义搜索空间：

  a. 使用autotune装饰器，以key-value形式由开发者定义搜索空间；

  b. 使用heuristics装饰器根据算子入参动态调整可调参数的搜索范围；

  模板库的寻优参数搜索空间也可采用类似的方式提供接口由开发者输入。

\3. 在搜索空间中寻优参数的每一个排列组合，为使其在代码中生效，需对算子代码中的寻优参数精准匹配并修改，修改后的临时代码完成运行时编译。

\4. 为评估算子运行性能，编译后的算子每次下发需完成性能评估，评估标准支持开发者配置。

\5. 完成寻优后打印输出最优参数与性能数据。

### 4.2.4 实现设计

#### 4.2.4.1 基础参数组合设计

可调参数通过模板参数集中传入算子，模板库算子一般需要对可调参数进行多次调整测试找到最优解，需要测试的参数组合即构成了参数寻优的搜索空间。

业界Triton通过对可调参数标记为”tl.constexpr”并通过搜索空间的配置接口以key-value形式支持开发者指定可调参数组合，具有较高的可读性，mskpp不妨也采用类似的设计，

\1.    在算子代码中标记可调参数，

\2.    通过搜索空间的配置接口以key-value形式定义可调参数的可选值，多个kv构成一个参数组合，多个参数组合构成寻优的搜索空间。

在遍历搜索空间寻优的过程中，需要考虑如下的关键功能点实现：

\1.    寻优参数的自动替换。替换寻优参数需要匹配需修改的参数所在代码并完成替换，参数匹配采用开发者输入的key值在算子代码中进行正则匹配，匹配对象限制在用 // tunable 标记的代码行。当需替换的代码段较长时，支持在标记后增加别名作为key值进行匹配，形如：// tunable: alias

\2.    自动编译。每次完成参数替换后，均需要重新进行编译流程，因编译过程独立且不会对运行性能造成影响，建议寻优时不同参数组合的参数替换与编译并行进行。

\3.    算子性能度量。每类参数组合测试时需要接入合适的工具对算子运行耗时进行度量，且需要考虑预热与波动影响，设置预热次数与重复次数。

\4.    搜索空间较大时，串行下发每个参数组合耗时上可能成本高昂，需要预留在多device上并行下发与度量的能力。

#### 4.2.4.2 支持模板库接入

待替换的目标参数替换，在开发者注释标记为 // tunable 或 // tunable: alias 形式的代码行进行，如下图组合所示，

![img](../pic/3751cfc8e2be8d7821b905510ab89033_660x142.jpg@900-0-90-f.jpg)

![img](../pic/9e46f19d34ab90f3d06375206a672f88_666x139.jpg@900-0-90-f.jpg)

支持两种模式进行匹配：

\1.    用key字符串参数去匹配开发者标记的别名，把标记代码行完整替换为value字符串，匹配优先级高于2；

\2.    用key字符串参数去匹配被标记的代码行，找到其被声明的代码行，=号右侧的实现完整替换为value字符串；

在不同作用域中，可能有两个同名的变量被声明，此时若两个变量均被标记为 // tunable 时，无法匹配需提示错误，建议开发者使用别名区分。

### 4.2.5 接口设计

如上一节所述，寻优参数的搜索空间参考Triton的设计，以key-value形式定义，通过autotune输入。

寻优参数的一个组合抽象作为一个dict，其中，每一个寻优参数及其取值抽象为一对key-value，多个组合组成list[dict]作为autotune接口的参数。

寻优时为使算子性能评估更为准确，需要提供预热、重复执行减少波动等能力，若算子输入输出的tensor在单次运行后会被改变，为避免其对算子性能测试产生的影响，还需要考虑增加每次寻优前tensor的恢复或置零的能力。在搜索空间较大时，为加速算子运行，考虑支持把算子并行下发到指定的device中。

表4-6 autotune接口参数

| **参数名**    | **说明**                                                     | **是否强制** | **入参/****出参/****返回值** | **默认值** |
| ------------- | ------------------------------------------------------------ | ------------ | ---------------------------- | ---------- |
| configs       | 寻优参数的配置组合                                           | Y            | 入参                         | /          |
| warmup        | 预热次数                                                     | N            | 入参                         | 0          |
| repeat        | 评估算子时的重复执行次数                                     | N            | 入参                         | None       |
| use_device    | 寻优测试任务并行分布至多卡进行，可指定多个device，如[0, 1, 2] | N            | 入参                         | None       |
| reset_to_zero | 每次测试前，指定tensor重置为0                                | N            | 入参                         | None       |
| restore_value | 每次测试前，指定tensor恢复默认值                             | N            | 入参                         | None       |

   kernel = get_kernel(src_path, kernel_name)
