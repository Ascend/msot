# **【检测】DETR网络整网精度问题定位案例**

## 一、问题背景

**问题描述**：整网直接拉起训练时，网络训练到一定step之后，网络中某个节点出现NAN，导致训练异常中断。

![image-20251212100930362](../pic/b71fb7983f3c7e374325a2577af109f9_2134x731.png)



## 二、定位思路

（1）怀疑是否是数据集、学习率、Dropout等导致NAN现象
（2）开关虚拟内存查看现象是否一致
（3）假设是内存问题，全面扫描网络中所有算子，排查有越界写问题的算子



## 三、定位步骤

**1、怀疑是否是数据集、学习率、Dropout等导致NAN现象**
多次更改torch dataloader随机数、学习率、Dropout比例，实验之后发现NAN现象出现的step均一致；故排除数据集、学习率、Dropout等的影响

**2、 开关虚拟内存查看现象是否一致**
开启虚拟内存之后，框架侧分配的内存是连续的，但可能会出现一些踩内存的现象。开关虚拟内存之后发现NAN出现的step一致，故和虚拟内存无关。

**3、假设是内存问题，使用预检工具+内存检测工具mssanitizer全面扫描网络中所有算子，排查有越界写入问题的算子**
扫描之后，有越界写问题的算子共3个：**DiagV2, DiagFlat, NLLLoss**

![image-20251212101045917](../pic/3e0bba5554843f700d364cf1d560b00a_1227x744.png)

![image-20251212101101851](../pic/ee525460b13cc54cfcf7b15cac46d386_1211x245.png)

![image-20251212101109637](../pic/a884a98d9e936e28b623be576cd465c8_1130x258.png)

将NLLLoss迁移至cpu之后NAN现象仍然存在，说明NAN现象和NLLLoss算子本身无关；
DiagV2为DiagFlat的反向算子，将对应的torch api **diag** 迁移至cpu之后，NAN现象消失，重点排查DiagV2和DiagFlat算子；
此外额外做了另外一个实验，将网络中的transpose算子迁移至cpu之后，NAN现象同样消失，但transpose算子本身并不存在相关内存问题。

基于以上实验现象，排查思路如下：

1. 怀疑是DiagV2 DiagFlat算子越界写踩踏了transpose算子，所以将diag 和transpose任意一个迁移至cpu之后，踩踏现象均不会发生
2. 联系对应算子开发工位，修复对应算子越界写问题
3. 由于目前预检工具dump的api存在遗漏问题，建议采集网络的profiling信息，拿到网络中DiagV2和DiagFlat算子的所有shape和dtype，算子工位内存问题修复好之后
     使用预检工具+内存检测工具分别验证算子精度问题和内存问题，都满足之后，将修复之后的算子.o替换cann包里的.o
   例如：预检工具dump的API存在如下[36, 36] [19]两种shape，但profiling额外多[51, 51]、[19, 19]两种shape:

4. 重新启动网络，查看NAN现象是否消失

修复好DiagFlat算子的越界写问题之后，NAN现象消失。

## 五、总结

整网内存问题排查基本思路

1. 先排除数据集、学习率等训练参数的影响
2. 全量扫描网络中所有算子内存问题，重点排查存在越界写问题的算子
3. 将存在越界写问题的算子迁移至cpu来排查越界写是否踩了其他算子地址空间
4. 重点排查迁移至cpu之后现象不一致的算子内存问题
5. 修复好之后使用预检工具+内存检测工具检查修复是否生效
6. 将修复好的算子替换原始cann里的算子，重新启动网络，观测现象
